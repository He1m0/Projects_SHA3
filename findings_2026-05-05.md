## 3) Why v1 F_9 Was Worse Than v2 F_9 — Root Cause

### Was this a bug or a deliberate modeling choice?

**It was a bug.** The evidence:

1. The F_9 model (You & Kuhn 2022 / Schindler 2005) explicitly requires
   per-position coefficients `c_l(t)` to be **fixed** properties of the
   hardware — they cannot change between evaluations of the same algorithm.
   A chip's transistor characteristics don't change per invocation.

2. The variable `invocation_sample_index` already existed in v1 and was
   already being reset to 0 correctly at every `KeccakP1600_leak_PermutationOnWords`
   call. The counter existed *for exactly this purpose*. Using `sample_index`
   instead was an oversight — the wrong variable was passed to the LUT lookup.

3. The v2 code comment makes the intent explicit:
   `# Use invocation_sample_index for F9 so all permutation invocations share`
   `# the same per-position coefficients (modular over the table size).`
   This is the behavior that was always intended.

**v2 is the correct implementation of F_9.** Stationary coefficients across
invocations (indexed by within-permutation position, not global trace position)
is what the F_9 model specifies and what real hardware exhibits.

### Short answer

v1 indexed F_9 coefficients by `sample_index` (global, accumulates across
all permutation invocations in a trace). v2 indexes by
`invocation_sample_index` (resets to 0 at the start of every Keccak-f[1600]
call). Because templates are fit by aggregating all invocations as independent
observations at the same within-permutation position, v1 fed the template
fitter a different regression model for every invocation. The LDA could not
converge on a stable function. v2 uses the same coefficients every time, so
the template sees a clean, consistent model.

### Code evidence

**v1 (`KeccakSim_BI_TA.py`, `_emit_sample_byte_pbw`):**
```python
contrib = self._pbw_byte_lut_at(self.sample_index)[int(byte_value) & 0xFF]
#                               ^^^^^^^^^^^^^^^^ global sample counter
```

`sample_index` is never reset between permutation invocations. It accumulates
across all invocations in a trace. If each invocation is L samples long,
invocation j at within-permutation position t uses coefficient row
`table[j·L + t]`.

**v2 (`KeccakSim_v2.py`, `_leak_signal`):**
```python
row = self.f9_table[self.invocation_sample_index % len(self.f9_table)]
#                   ^^^^^^^^^^^^^^^^^^^^^^^^^ reset to 0 at each invocation
```

`invocation_sample_index` is reset to 0 at the start of every
`KeccakP1600_leak_PermutationOnWords` call. At within-permutation position t,
every invocation, every trace, uses `table[t % T]`.

### Why this kills template quality

The training pipeline combines `SHA3_INVOCATIONS=10` invocations per trace
into a single template at each trace position t. Template fitting assumes the
relationship `trace[t] = f(byte_value)` is stationary across all observations
at position t.

Under v1: invocation j contributes observation with coefficient row
`table[j·L + t]` — a different set of F_9 coefficients for each j. The LDA
sees a mixture of 10 different linear models at the same position. Aggregating
these is equivalent to fitting a model whose signal is averaged out. SNR
collapses to near-HW levels.

Under v2: every invocation at position t uses `table[t % T]`. All 10
invocations contribute observations drawn from the same linear model. The LDA
fit is well-posed. Templates become near-perfect (A-family SR 99.7%).

### Physical interpretation

The bug simulates a physically impossible chip: one where the circuit's
leakage function changes with every evaluation of the algorithm. Real hardware
has fixed transistor characteristics — the same computation at the same point
in the permutation leaks with the same coefficients every time. v2 is
physically correct; v1 was not.

### Why the collapsed byte-pbw escaped this problem

The collapsed model draws one shared 8-vector at init time. Because there is
only one vector (not a per-PoI table), `sample_index` vs `invocation_sample_index`
doesn't matter — both would return the same constant vector. The collapsed model
"accidentally" had the right invocation-invariant property. This is why the
theoretically weaker collapsed model outperformed the theoretically correct
F_9 v1 in all previous smoke runs.

## 4) What These Results Mean

### The attack works

v2 F_9 in self-consistent mode (simulate with F_9, fit templates with LDA →
LDA learns F_9 structure) gives:
- A-family SR 99.7% with GE ≈ 1 (templates are essentially perfect)
- 100% SASCA key recovery at all 21 rate points, all round depths

This is above the real-silicon reference (A=35.7%). The pipeline is sound —
given a leakage model that matches reality, the attack succeeds completely.

### The remaining gap is leakage characterization

Real silicon does not have perfectly stationary F_9 coefficients with
`U(0,1)` draws. The real SR of ~35.7% suggests the actual leakage is weaker
or noisier than our idealized F_9 simulator. The gap is now characterized as:

> "How accurately can we estimate the real hardware's per-position F_9
> coefficients from oscilloscope data?"

This is the You & Kuhn 2022 profiling problem — it is the problem the paper
addresses with their measurement campaign.

### HD is not the missing piece

v2 HW+HD (scale=0.5) scores A=4.1% — *slightly worse* than v2 HW at 4.4%.
The CMOS transition-power hypothesis is rejected as a major contributor to the
sim-vs-real gap, at least at this additive scale. The F_9 coefficient structure
(per-bit, per-position) is the dominant physics, not HD.

### The v1 F_9 anomaly is resolved

The counter-intuitive result — theoretically correct F_9 is worse than
theoretically wrong collapsed pbw — is explained fully by the indexing bug.
It was an implementation artifact, not a statement about the F_9 model.

## 6) Decision Retrospective

Looking back at the causal chain that led here:

1. **Apr 29 v1 F_9** was implemented correctly at the formula level but had a
   subtle indexing bug that made it appear worse than the collapsed model.
2. The seed sweep (Apr 30) and signed-coefficient runs correctly ruled out
   coefficient distribution as the cause. But they couldn't distinguish
   "F_9 is wrong physics" from "F_9 is correctly implemented but bug degrades
   it" — both would produce the same symptom.
3. The additive HD runs (May 1–3) provided a real (if partial) improvement,
   confirming that *some* new physics term can help — but the scale of
   improvement was limited by the same v1 bug underlying the F_9 baseline.
4. The **v2 rewrite** (May 4) fixed the bug as a side effect of restructuring
   the simulator around `invocation_sample_index`. The fix was not targeted at
   this specific bug — it was a clean-room reimplementation.

The right retrospective question: could we have found the bug earlier?
Yes — a unit test comparing `trace[t_in_invoc_0] ≈ trace[t_in_invoc_1]`
(in noise-free mode, same input byte) would have revealed that v1 F_9
produces different signals for the same byte at the same within-permutation
position across invocations. Adding this invariant check to the simulator
test suite is recommended before any future model comparisons.
