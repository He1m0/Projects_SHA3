## 3) SNR Analysis — How It Is Calculated and What the Sweep Means

### 3.1 What "SNR" means in template attacks

In side-channel analysis, SNR is defined as a **variance ratio**, not an amplitude
ratio. The standard definition (Mangard et al., "Power Analysis Attacks", 2007):

```
SNR = Var(signal) / Var(noise)
    = between-class leakage variance / measurement noise variance
```

This is distinct from the amplitude SNR (signal_std / noise_std) used in RF/audio —
the SCA variance definition matters because template attacks are Maximum Likelihood
classifiers operating on variance structure, not amplitude thresholds.

### 3.2 Computing Var(signal) for our F₉ model

The F₉ leakage at one sample point t is (Schindler, Lemke, Paar 2005):

```
l(t) = Σᵢ₌₀⁷ bᵢ · cᵢ(t)  +  c₈(t)  +  ε(t)
         ───────────────────    ──────    ────────
         byte-dependent signal  DC offset   noise
```

- `bᵢ` = bits of the target byte, each Bernoulli(0.5) → Var(bᵢ) = 0.25
- `cᵢ(t)` = F₉ bit coefficients, fixed per circuit position; in `KeccakSim_v2.py`
  drawn from **U(0, 1)** (not symmetric). The SIM_F9_C8_RANGE=0.5 parameter only
  controls the intercept c₈, **not** the bit coefficients.
- `c₈(t)` = constant offset drawn from U(−0.5, +0.5), **does not contribute to
  between-class discrimination** (same for all input classes)
- `ε(t) ~ N(0, σ²)` = measurement noise

The **discriminating signal** is only the byte-dependent part. Its between-class
variance (the quantity that makes classification possible) is:

```
Var_b[Σᵢ bᵢ·cᵢ] = Σᵢ cᵢ² · Var(bᵢ)        (bits are independent)
                  = 0.25 · Σᵢ cᵢ²
```

Averaging over the random draw of coefficients.
For cᵢ ~ U(0, 1): E[cᵢ²] = ∫₀¹ x² dx = 1/3.

```
E[Var_signal] = 8 · (1/3) · 0.25 = 2/3 ≈ 0.667
signal_std per PoI ≈ √0.667 ≈ 0.816
```

This is the expected signal variance for a single sample point. Individual PoIs will
vary (cᵢ are random), but the average is 0.667.

**Note:** An earlier calculation (now corrected) used cᵢ ~ U(−0.5, +0.5) giving
E[cᵢ²]=1/12 and E[Var_signal]≈0.167. This was 4× too low. The correct E[cᵢ²]=1/3
because the simulator draws from U(0,1), not a symmetric distribution.

### 3.3 The SNR table

```
SNR_var = Var(signal) / Var(noise) = 0.667 / σ²
```

| σ | SNR_var | Amplitude ratio (signal_std / σ) |
|---|---|---|
| 0.0007 | 1,360,000 | 1,166 |
| 0.01 (pipeline default) | 6,670 | 82 |
| 0.5 | **2.67** | **1.63** |
| 1.0 | **0.667** | **0.816** |
| 1.5 | **0.296** | **0.544** |
| 2.0 | **0.167** | **0.408** |
| 2.5 | **0.107** | **0.327** |
| 3.0 | **0.074** | **0.272** |
| 4.0 | **0.042** | **0.204** |

**SNR_var first falls below 1 between σ=0.5 and σ=1.0** (crossover at σ=√0.667≈0.816).
Every previous pipeline run (σ=0.0007 or σ=0.01) was operating at SNR_var ≫ 1 —
effectively noise-free for template discrimination.

### 3.4 Comparison with the scale sweep (why that one was easy)

The May 6 scale sweep reduced F₉ amplitude 10× (F9_SCALE=0.1) at fixed σ=0.0007.
The effective signal variance scales as scale²:

```
Var_signal(scale=0.1) = 0.1² × 0.667 = 0.00667
SNR_var = 0.00667 / (0.0007)² = 0.00667 / 4.9×10⁻⁷ ≈ 13,600
```

Thirteen thousand times above unity. The attack never touched its noise floor in the
scale sweep. The sigma sweep is the **first experiment that actually tests noise
robustness** — all prior sweeps (scale, HD mixing, ICS level) were at astronomical
SNR.

### 3.5 Should this sweep have been expected to fail?

**At smoke scale (50 SASCA traces), yes.** But the failure is scale-dependent,
not fundamental. Template attacks have been demonstrated at SNR_var < 0.01 in the
literature (e.g. Batina et al. 2014, Durvaux & Standaert 2016) — but only with
hundreds to thousands of attack traces, which allow test-time averaging.

The F₉ regression-derived template is already well-estimated at smoke scale. The
coefficient estimation error at σ=1.0 is:

```
σ / √N_train = 1.0 / √8,000 ≈ 0.011
Typical |cᵢ| ≈ 0.5  →  relative error ≈ 2.2%
```

Templates are not the bottleneck. The bottleneck is **test-time SNR**: a single
noisy trace at SNR_var=0.667 cannot reliably identify 1 key byte from 256 candidates.
Paper scale with 1,000 SASCA test traces allows BP to aggregate evidence across many
weak single-trace LLRs — this is where we expect the improvement.

### 3.6 Does the sweep make sense? Is there a mistake in the noise model?

**The sweep is correct and the noise model is standard.** Additive Gaussian noise
per sample point is the canonical stochastic leakage model (Schindler–Lemke–Paar
2005). No implementation bug.

The potential confusion is a **normalization mismatch**: in our simulator the F₉
bit coefficients are U(0,1) giving signal_std ≈ 0.816. If real hardware coefficients
were different in scale, the same physical noise would correspond to a different
relative σ. The sweep characterizes noise robustness under one normalization; the
absolute σ values cannot be compared to oscilloscope noise in volts without
independently estimating the real hardware's coefficient magnitudes (which is exactly
the profiling step in You & Kuhn 2022). A normalization target of E[Var_signal]=1.0
would require scaling coefficients by √(3/2)≈1.2247, but this has not been
implemented — all current runs use the unnormalized U(0,1) draw.

Concretely, the sweep from σ=0 to σ=4 covers:

| σ range | SNR_var range | Physical interpretation |
|---|---|---|
| 0–0.1 | ≥ 667 | Near-ideal; comparable to our realnoise default |
| 0.5 | 2.67 | Strong signal; equivalent to well-shielded clean-lab |
| 0.5–0.8 | 2.67–1.0 | Signal-noise crossover: SNR_var crosses 1 at σ≈0.816 |
| 1.0–1.5 | 0.667–0.296 | Realistic unprotected ARM Cortex-M measurement |
| 2.0–2.5 | 0.167–0.107 | Challenging measurement (long cables, higher noise) |
| 3.0–4.0 | 0.074–0.042 | Very noisy / poorly decoupled target |

**All of these regimes are physically meaningful.** Real SCA operates across this
entire range depending on measurement quality. The supervisor's sweep is the right
experiment — it maps out the noise tolerance boundary. The "expected failure" at
σ≥1 at smoke scale is an artifact of having only 50 SASCA traces, not a statement
about the fundamental limit.

### 3.7 What paper scale can and cannot fix

**Cannot fix:** first-order SR per trace. At SNR_var=0.667, a single test trace
cannot reliably distinguish 256 byte hypotheses regardless of training set size.
First-order SR is expected to improve only modestly (if at all) vs smoke scale.

**Can fix:** SASCA. 1,000 SASCA traces (vs 50 at smoke) give belief propagation 20×
more evidence to aggregate. At SNR_var=0.667, the per-trace LLR contribution is weak
but non-zero; summing 1,000 of them may push BP past the convergence threshold.
This is the primary question the paper-scale sigma sweep will answer.

### 3.8 Comparison with real hardware SNR

| σ in our sim | SNR_var | Comparable hardware regime |
|---|---|---|
| 0.01 (default) | 6,670 | Laboratory ideal |
| 0.5 | 2.67 | Strong signal; well-shielded ChipWhisperer measurement |
| 0.816 | 1.00 | Signal = noise (crossover point) |
| 1.0 | 0.667 | Typical unprotected ARM Cortex-M target |
| 2.0 | 0.167 | Noisy or high-frequency target |
| 3.0+ | 0.074 | Poorly shielded / RF-interfered measurement |

### 3.9 What the paper's [0.01, 3.43] SNR range means

You & Kuhn 2022 (p.15) report: "The SNR values of the points selected were in the
range 0.01 to 3.43." This is **per-sample-point SNR** across all time positions that
survived ICS selection — it describes heterogeneity in the trace, not overall attack
quality.

Real hardware traces are highly heterogeneous: some clock cycles (e.g. S-Box
transitions) produce strong leakage (SNR≈3.43), while most selected positions are
near the R² selection threshold (SNR≈0.01). The range [0.01, 3.43] is a one-sided
distribution heavily skewed toward the low end.

**Comparison to our simulator:**

Our F₉ model also produces per-PoI SNR variation (because coefficients are drawn
randomly per position), but the distribution shape differs fundamentally:

| Configuration | PoI SNR range | Median PoI SNR | Distribution shape |
|---|---|---|---|
| Our sim σ=0.5 | [0.107, 6.532] | 2.63 | Near-symmetric around mean |
| Our sim σ=1.0 | [0.099, 1.633] | 0.658 | Near-symmetric around mean |
| Our sim σ=2.0 | [0.099, 0.408] | 0.171 | Compressed range |
| Paper real HW | [0.01, 3.43] | << median | Heavy right tail, mostly near threshold |

Key insight: the paper has a **few very strong PoIs** (SNR≈3) among **many weak
PoIs** (SNR≈0.01). Our simulator has all PoIs contributing approximately equally
(same U(0,1) coefficient distribution everywhere). This is a structural difference
between uniform random coefficients and real hardware physics.

The sigma value where our strongest PoI matches the paper's maximum (3.43):
σ≈0.69. But this doesn't mean σ=0.69 is the "correct" matching point — the
overall distribution shapes are different regardless of σ. The sweep from σ=0.5–4.0
correctly spans all physically relevant noise regimes without needing to match
the paper's exact per-PoI distribution.

The sigma sweep therefore maps directly onto the range of real-world measurement
quality. Every previous pipeline run at σ≤0.01 (SNR_var≥6,670) was operating far
outside realistic hardware conditions.

## 4) The σ=3.0–4.0 Identical-Results Anomaly

All three runs (σ=3.0, 3.5, 4.0) return byte-identical SR/GE metrics and identical
Rate_Scan per-trace histograms (n=132, min=5, median=12, max=20). This is suspicious.

Most likely cause: at these extreme noise levels, the ICS detection (0002) fails to
find any sample points exceeding the R² thresholds, causing the pipeline to fall back
to a noise-driven degenerate state. If the ICS archive is empty or minimal, the
template profiling and BP stages produce arbitrary but reproducible outputs (seeded
RNGs). The outputs reflect the seed, not the noise level.

This should be verified by checking the ICS archive sizes and R² distributions for
the σ=3.0–4.0 runs. If confirmed, these runs are not measuring noise robustness —
they are measuring pipeline failure mode behavior.

### 5.2 Parameter changes vs smoke sigma sweep

| Parameter | Smoke | Paper | Reason |
|---|---:|---:|---|
| SHA3_TRAINING_SET_COUNT | 50 | 400 | 8× more training data |
| SHA3_VALIDATION_SET_COUNT | 20 | 40 | Consistent with paper scale |
| SHA3_SASCA_TRACE_COUNT | 50 | 1,000 | 20× more SASCA test traces — primary gain |
| SHA3_SASCA_RATE_POINT_COUNT | 21 | 201 | Higher-resolution Rate_Scan curve |
| SHA3_SASCA_RATE_STEP_BITS | 64 | 8 | Finer granularity (512/201 ≈ 2.6 b/step) |

### 5.3 Expected outcomes

| σ | Expected 1st-order A-SR | Expected SASCA improvement |
|---|---|---|
| 1.0 | 5–15% (modest uplift from better template est.) | Likely significant; 20× more traces |
| 1.5 | 3–8% | Possible improvement at max trace count |
| 2.0 | 2–5% | Marginal; SNR_var=0.04 is very low |
| 2.5 | Near baseline (~1.4%) | Unlikely to help much |
| 3.0+ | Near/at baseline (~1.4%) | Paper scale probably cannot fix; ICS failure |

If first-order SR is unchanged (smoke ≈ paper) at all sigma values, this confirms
that template estimation is already saturated at smoke scale and the failure is
purely test-time SNR limited. That result itself is informative.
